## 🧭 1. 프로젝트 오너(송훈)가 자동화하려는 것

### 현재 수동으로 하고 있는 주간 리서치 워크플로우

매주 오너가 손으로 직접 하는 작업:

1. NotebookLM에 축적된 강의 자료(기반 노트북)에서
   5개 분야별로 "이번 주 찾아볼 검색식"을 요청
2. 그 검색식으로 지난주 뉴스를 조사
3. 검색 결과를 NotebookLM Deep Research로 돌려
   소스 수집 + 새 노트북 생성
4. 완성된 노트북에서 5가지 유형의 인사이트 추출:
   - 이번 주 패러다임 시프트 (구조적 변화)
   - 이면의 시그널 (표면 뒤 숨은 의미)
   - 밸류체인 파급 (산업 연쇄 효과)
   - 자본 흐름 나침반 (방향성 읽기, 단기 트레이딩 팁 아님)
   - 산업 지형도 변화 (2~3개 뉴스 엮어 만드는 빅 인사이트)
5. 이 중 마음에 드는 것을 골라 유튜브 에피소드 소재로 활용

### 자동화 목표

이 5단계를 NerdyHub 웹앱에서 관리자가 버튼 몇 번으로 처리할 수 있게 만든다.
완전 자동화가 아니라 **Human-in-the-loop** — AI가 초안을 만들고, 오너가 검수·선택·발행.

## 🗄️ 4. DB 설계 결정 이유

### 왜 INT[]가 아니라 TEXT(JSON 문자열)인가

news_insights.related_news_ids 컬럼은 여러 뉴스를 묶는 크로스 인사이트에 필요.
PostgreSQL의 INT[] 배열 타입은 SQLAlchemy ORM에서 매핑이 번거롭다.
이 프로젝트가 ORM을 쓰는지 raw psycopg2를 쓰는지 확인 전에는 알 수 없으므로,
어느 쪽에서도 동작하는 TEXT(JSON 문자열) 방식을 선택.

```python
# 저장
json.dumps([1, 3, 7])  →  '[1, 3, 7]'

# 읽기
json.loads('[1, 3, 7]')  →  [1, 3, 7]
```

헬퍼 함수 parse_related_ids() / serialize_related_ids()를 app.py에 추가할 것.

### 왜 full_text 컬럼을 지금 만드는가

3.9-D에서는 Google RSS snippet만 수집하지만,
3.9-E에서 newspaper3k 또는 BeautifulSoup으로 기사 본문 전체를 수집할 예정.

**이유:** RSS snippet만으로 Gemini가 인사이트를 생성하면 할루시네이션 위험이 크다.
"이면의 시그널"이나 "산업 지형도 변화" 같은 깊이 있는 분석은 본문 전체가 필요.

3.9-D에서는 full_text = NULL로 저장.
3.9-E에서 UPDATE news_raw SET full_text = ... WHERE id = ... 패턴으로 채운다.
컬럼을 지금 만들어두면 나중에 ALTER TABLE 없이 바로 사용 가능.

- Railway 서버 환경에서 쿠키 인증 관리가 복잡
- 3.9-D~F는 MCP 없이도 완전히 동작해야 함


MCP가 없어도 Gemini API만으로 전체 파이프라인이 돌아가야 한다.
