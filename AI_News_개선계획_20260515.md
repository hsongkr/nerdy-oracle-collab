## 1. 현황 진단

### 1-A. 현재 파이프라인 구조 (3.9-D/E 완료 상태)

```
Google News RSS (feedparser)
  └→ news_raw 저장 (full_text 최대 5,000자 크롤링)
       └→ Gemini 인사이트 생성 (각 기사 500자만 전달)
            └→ NewsInsight(draft) 저장
                 └→ 수동 검수 → published
                      └→ /news-analytics 표시
```

### 1-B. 확인된 품질 문제

| 문제 | 원인 | 영향 |
|------|------|------|
| 분석이 얕음 (표면적 사실 나열) | 기사 본문 500자 제한 → 맥락 소실 | 인사이트 body가 피상적 |
| 뉴스 필터링 없음 | 최근 10건을 그냥 전달 | 어그로/저품질 기사가 인사이트에 영향 |
| 인사이트 유형 정의 불명확 | paradigm_shift vs hidden_signal 기준 없음 | 유형 분류가 랜덤에 가까움 |
| body 분량 부족 | 프롬프트에 길이 가이드 없음 | 블로그 글 수준에 못 미침 |
| 시황 보고서도 얕음 | 뉴스 5건만 참고, 분석 프레임워크 부재 | 섹터별 맥락 없는 단순 번역 수준 |
| DB 성장 우려 | full_text 컬럼이 기사마다 최대 5KB | 누적 시 Railway PostgreSQL 비용 ↑ |
| 토큰 비용 우려 | 전문 분석 시 Gemini 유료 사용량 급증 예상 | 운영 비용 통제 필요 |

## 2. 개선 옵션 분석

### Option A: 현행 아키텍처 + 프롬프트 강화 (저비용·빠름)

**변경 내용:**
- 기사 전달 컨텍스트 500자 → 1,500자로 확대
- Gemini 프롬프트에 분석 프레임워크 + 인사이트 유형별 정의 + 길이 가이드 추가
- 뉴스 10건 중 Gemini Flash로 1차 중요도 점수화(랭킹) 후 상위 5건만 Pro에 전달

**장점:**
- 코드 변경 최소 (프롬프트 수정 + 컨텍스트 길이 조정)
- 추가 의존성 없음
- 토큰 비용 증가 제한적

**단점:**
- 크롤링 실패한 기사(full_text=None)는 여전히 품질 한계
- 랭킹 로직 자체가 Gemini 호출 → 1건 추가 API 비용

**예상 품질 향상:** ⭐⭐⭐☆☆

---

### Option B: 2단계 Gemini 파이프라인 (중간 비용·높은 품질)

**변경 내용:**
```
[STEP 2] RSS 수집 + BeautifulSoup 크롤링 (현행 유지)
   ↓
[STEP 2.5 신규] Gemini Flash — 뉴스 중요도 분류
   · 입력: 기사 제목 + 본문 1,000자
   · 출력: 중요도 점수(1-5) + 분류 태그
     (예: "산업파급력=5, 어그로성=1, 트렌드대변=4, 주가영향예상=3")
   · 비용: Flash 모델(저가) 사용으로 절감
   ↓
[STEP 3] Gemini Pro — 상위 5건으로 심층 인사이트 생성
   · 입력: 기사 전문(최대 3,000자) + 중요도 점수 + 태그
   · 출력: 블로그 포스트 수준 body (500-1,000자)
```

**장점:**
- 중요 뉴스만 Pro 모델에 전달 → 비용 효율적
- 1차 분류로 어그로/저품질 기사 자동 제거
- 사용자가 원하는 뉴스 유형(중요도/주가영향/트렌드) 태깅 가능

**단점:**
- Gemini API 호출 2회 → 레이턴시 증가 (약 30-60초)
- STEP 2.5 프롬프트 설계 필요
- 중요도 판단 기준은 여전히 AI 의존

**예상 품질 향상:** ⭐⭐⭐⭐☆

### Option C: NotebookLM MCP 연동 (고품질·반자동)

**전제:** Anti Gravity가 이미 MCP 서버 환경 준비 완료 (`@jacob-bd/notebooklm-mcp-cli`)

**변경 내용:**
- 수집된 기사 URL → NotebookLM 노트북에 소스로 자동 추가
- NotebookLM의 긴 문서 통합 분석 능력 활용
- 인사이트는 NotebookLM에서 생성 후 수동 또는 반자동 복사

**장점:**
- NotebookLM의 소스 통합 분석이 현재 단순 Gemini 호출보다 정교
- 긴 컨텍스트 처리 가능 (기사 원문 전체)

**단점:**
- **자동화 어려움**: NotebookLM MCP는 현재 CLI 기반 → Webhook/API 자동 트리거 불가
- **운영 흐름 이탈**: 기존 파이프라인(admin_news_pipeline) 밖에서 작동
- **3rd-party 의존성**: Google의 서비스 정책 변경에 취약

**예상 품질 향상:** ⭐⭐⭐⭐⭐ (but 반자동, 운영 비용 예측 어려움)

**단점:** 뉴스 컨텍스트 품질(건수/길이)은 여전히 제한

## 4. 프롬프트 개선 방향 상세 (Option A 기준)

### 4-A. 인사이트 생성 프롬프트 개선 요소

**현재 프롬프트의 문제:**
```
You are an expert tech and financial analyst (Nerdy Oracle).
Based on the following recent news for the category '...', generate 1 or 2 deep-dive insights.
...
"Detailed markdown body explaining the insight (in Korean)..."
```

**개선 방향:**

1. **인사이트 유형별 명확한 정의 추가**
```
Insight Type Definitions:
- paradigm_shift: 기존 산업/기술 패러다임을 뒤집는 구조적 변화. 예) AI가 소프트웨어 개발자 직군을 대체하기 시작한 증거
- hidden_signal: 표면적으로 작아 보이지만 향후 큰 영향을 줄 이면의 신호. 예) 대형 클라우드 3사가 조용히 AI칩 자체 개발을 가속화하는 패턴
- value_chain: 공급망/밸류체인 재편으로 수혜/피해 받는 기업군 변화. 예) 엣지 AI 확산으로 전통 클라우드 수요 구조 변화
- capital_compass: 스마트 머니(기관투자자, VC, 정부)가 어디로 흐르는지. 예) 미국 국방부의 AI 예산 배정 패턴 변화
- landscape_shift: 플레이어 간 경쟁 구도 변화. 예) OpenAI vs Google의 기업 AI 계약 전쟁
```

2. **분석 프레임워크 (What → Why → So What) 추가**
```
각 인사이트의 body는 다음 구조를 따를 것:
1. 핵심 사실 (What): 무슨 일이 일어났나 (1-2문장)
2. 구조적 원인 (Why): 왜 이것이 의미 있나, 산업 맥락
3. 파급 효과 (So What): 이것이 앞으로 어떤 변화를 만들 것인가
- 분량: 500-800자 (블로그 포스트 한 단락 수준)
- 소제목(##), 굵게(**), 핵심 숫자 포함 권장
```

3. **컨텍스트 확대: 500자 → 1,500자**

### 4-B. 시황 보고서 프롬프트 개선 방향

**추가할 요소:**
- Nerdy Oracle 톤앤매너 명시: *"전문 리서치 애널리스트처럼 분석적으로, 투자 결정을 돕는 인사이트 중심으로 작성. 단순 뉴스 번역이 아닌 '왜 이것이 오늘 시장에 중요한가'를 설명."*
- 섹션별 분량 가이드: I. 200자 / II. 250자 / III. 150자
- 데이터 활용 강화: *"지수 숫자를 직접 인용하고, 전일 대비 변동폭의 의미를 해석할 것"*
- 금지 표현 강화: "~ 할 것 같습니다" (불확실 표현) 지양, "~로 판단됩니다" 사용 권장

## 5. DB 및 비용 리스크 관리

| 항목 | 현황 | 개선 제안 |
|------|------|----------|
| news_raw.full_text | 기사당 최대 5,000자, 무기한 보존 | 수집 후 30일 지난 기사 full_text NULL 처리 (텍스트만 삭제, 메타데이터 유지) |
| Gemini 토큰 사용량 | Flash 미사용, Pro만 사용 | 1차 분류에 Flash 사용, Pro는 최종 인사이트만 |
| RSS 수집 빈도 | 수동 트리거 | 현행 유지 (자동화는 Phase 4에서 검토) |
| news_raw 보존 정책 | 없음 | 발행된 인사이트와 연결된 기사만 장기 보존 |
