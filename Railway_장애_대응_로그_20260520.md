#### 2-D. 최근 커밋 코드 변경사항 검토
- `_gemini_generate()` 함수에 `logger.warning()` 추가 → `logger`가 line 71에 정의, 함수는 line 37 ⚠️ 의심
  - 분석 결과: Python은 함수 호출 시점에 전역 변수를 resolve → **문제 없음** ✅
- `time.sleep(wait)` 추가 (10s, 20s) → 단일 워커 블로킹 가능성 ⚠️ **실제 위험 요소 발견**

#### 2-E. Gunicorn 단일 워커 블로킹 문제 발견 및 수정
**문제**: `Procfile: gunicorn --timeout 120 -w 1`
- 워커가 1개인데 `time.sleep(30)`이 포함되면 Gemini 503 재시도 중 모든 요청 block
- Railway 헬스체크 실패 → 서비스 다운처럼 보일 수 있음

**수정 (commit: `437f344`, 2026-05-19 23:52 UTC)**:
```python
# 변경 전: 10s, 20s sleep 후 재시도
time.sleep(10 * (2 ** attempt))

# 변경 후: sleep 없이 즉시 재시도
continue  # sleep 제거
```

**그러나 이 수정 후에도 Railway 접속 불가 상태 지속** → 코드 문제가 아님을 확인

## 4. 학습 사항 (재발 방지)

### 4-1. 접속 불가 시 진단 순서 (개선된 체크리스트)

```
① status.railway.app 먼저 확인 ← 가장 빠른 진단
② Railway 대시보드 접속 가능 여부 확인
   - 불가 → 플랫폼 장애 (코드 점검 불필요)
   - 가능 → 내 서비스 로그 확인
③ 로그에서 Python traceback 여부 확인
   - 있으면 → 코드 문제
   - 없으면 → 인프라/설정 문제
```

### 4-2. 코드 관련 실제 수정 사항 (장애와 별개)

**commit `437f344`**: Gunicorn 단일 워커 환경에서 `time.sleep()` 제거
- `-w 1` 환경에서 30초 sleep은 모든 요청 블로킹 유발 가능
- 앞으로 Railway 단일 워커 환경에서는 장시간 sleep 사용 금지
- 비동기 처리가 필요하면 워커 수 증가(`-w 2`) 또는 background thread 사용

### 4-3. Railway 플랫폼 의존도 리스크

- Railway는 Google Cloud 위에서 운영 → Google Cloud 장애 시 동반 다운 위험
- 이번 사례: Google Cloud가 Railway **계정 자체를 차단** (정책 위반 또는 오탐)
- 미래 대안 고려: Render, Fly.io, Heroku 등 멀티 클라우드 백업
